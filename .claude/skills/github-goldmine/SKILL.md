# GitHub 变现机会挖掘(github-goldmine)

把 GitHub 当作公开的需求库:官网只说产品多好,GitHub 会留下不好看的真相——维护状态、用户卡点、没人接的需求。本 skill 的任务是把这些真相变成**可交付的服务机会**,并沉淀进台账持续积累。

## 核心立场(先读,贯穿全流程)

1. **卖服务和结果,不卖文件。** 倒卖免费源码包(9.9 资源包模式)门槛低、侵权风险高、售后崩坏,本 skill 不产出这类建议。有价值的机会形态是:代部署、汉化、教学、行业模板、数据迁移、托管维护、免费引流+付费完整方案。
2. **License 是红线,不是脚注。** 每个机会必须标注许可证及其商用含义。NC(非商用)类、无 License 的项目直接排除出机会列表;GPL 系需注明"衍生代码必须开源,卖服务不受限";MIT/Apache 需注明保留版权声明。
3. **不做平台自动上架。** 闲鱼等平台的自动化发布违反平台规则,本 skill 只产出调研结论和人工操作清单,不执行上架。

## 输入与两种模式

- **领域模式**(默认):用户给一个领域/需求关键词(如"个人记账"、"播客转文字")→ 跑完整五阶段流水线。
- **单仓模式**:用户给一个具体 repo(URL 或 owner/name)→ 只跑阶段 2(五分钟判断法)+ 阶段 3(需求挖掘),输出该项目的"直接试/需要技术基础/观望/关掉"结论;若发现服务机会,同样写入台账。

用户输入含糊时(如只说"帮我挖挖机会"),先问清领域再开跑;领域太宽(如"AI")时主动收窄成 2-3 个候选细分,让用户挑。

## 数据沉淀位置

所有持久化数据放在当前工作目录(或 git 仓库根目录)的 `github-goldmine-data/` 下:

```
github-goldmine-data/
├── ledger.json          # 机会台账(结构化,跨次运行累积)
├── opportunities.md     # 台账的人类可读版(每次运行后重新生成)
└── reports/
    └── YYYY-MM-DD-<领域>.md / .html   # 每次运行的报告
```

运行开始时先读 `ledger.json`(存在的话):已记录的机会不重复报告,但要检查其状态是否需要更新(如项目已归档 → 标记"已失效")。台账结构见 `references/templates.md`。

## 环境探测(阶段 0)

GitHub 访问方式按顺序探测,用第一个可用的:

1. **`scripts/scan_github.py`**(优先):`python3 scripts/scan_github.py --query "..." --output candidates.json`。走 api.github.com,支持 `GITHUB_TOKEN` 环境变量提升配额。脚本输出含 `"restricted": true` 或报错 → 降级。
2. **`gh` CLI**(如果已安装且登录):`gh search repos`、`gh api repos/...`。
3. **GitHub MCP 工具**(如果会话有且未限定仓库范围)。
4. **WebSearch + WebFetch 兜底**:用 WebSearch 搜 `site:github.com <关键词>` 或直接搜领域词,再 WebFetch 各 repo 主页 / releases / issues 页面提取信息。受限环境(如仓库范围锁定的云会话)下这条路始终可用,只是每仓成本高,候选控制在 8-12 个。

市场竞争调研的环境探测见 `references/competition.md`(浏览器 → 网络搜索 → 人工清单三层降级)。

## 五阶段流水线(领域模式)

### 阶段 1:候选扫描

把领域词展开成 3-5 组搜索(不要只搜一个词):

- `<关键词> stars:>100 pushed:>去年今日 archived:false`(主流工具)
- `<关键词> alternative` / `<关键词> self hosted`(替代品需求聚集地)
- `<关键词> gui` / `desktop app` / `中文`(最后一公里缺口信号)

收集 10-20 个候选,记录:名称、描述、stars、最近推送、license、是否有 Releases 成品。**这一步宽进,别急着判断。**

### 阶段 2:五分钟判断法批量筛选

对每个候选按 `references/scoring.md` 的四查打分(README / Releases / 维护状态 / Issues),得出四选一结论:**直接试 / 需要技术基础 / 观望 / 关掉**。

关键认知:「需要技术基础」不是淘汰信号,而是**机会信号**——项目有价值但普通人装不上,正是"最后一公里"服务的位置。筛选后保留 5-8 个进入下一阶段:优先"需要技术基础"和"直接试但配置复杂"的项目。

### 阶段 3:需求挖掘(Issues 考古)

对保留的项目逐个挖 Issues(方法细节和信号强度判断见 `references/scoring.md` 后半部分):

- 搜关键词:安装、install、windows、docker、中文、export、导出、同步、教程、tutorial
- 记录:同一抱怨出现了几次、维护者回应(修复/拒绝/wontfix/沉默)、用户有没有自己写脚本绕过(付出成本 = 强信号)
- **跨项目聚合**:同一类抱怨在 3 个以上项目重复出现 → 这是一类人的共同麻烦,标记为高价值需求

维护者明确 wontfix 但用户持续追问的需求,是最好的缺口——原作者不做,不代表没人愿意付钱。

### 阶段 4:竞争调研

按 `references/competition.md` 执行:有浏览器工具就直接查闲鱼/淘宝在售情况;没有就用 WebSearch 测内容饱和度;无论哪层,**都要**在报告里附人工验证清单(10 分钟人工抽查比爬到的数据准)。

### 阶段 5:机会成型与输出

每个候选机会过"机会四问"(任何一问答不上就降级或放弃):

1. 用户具体卡在哪一步?(要有 Issues 证据链接)
2. 我能不能把这一步做成**明确交付**?(一句话说清:"帮 Windows 用户 10 分钟装好 X")
3. 买家拿到的是可用结果,还是一堆文件?
4. 原项目更新后,我愿不愿意持续维护?

产出三件套(格式见 `references/templates.md`):

1. **Markdown 报告** → `reports/YYYY-MM-DD-<领域>.md`:候选全景表、机会详情(证据、服务形态、定价建议、风险)、人工验证清单
2. **台账更新** → 新机会以状态 `待验证` 追加进 `ledger.json`,重新生成 `opportunities.md`
3. **HTML 可视化** → 用 `assets/report_template.html` 模板,把报告数据 JSON 注入 `__GOLDMINE_DATA__` 占位符,输出 `reports/YYYY-MM-DD-<领域>.html`。有 Artifact 工具就同时发布;没有就发送文件给用户。

## 定价建议的锚点

不要拍脑袋报价,按交付形态给区间:

| 交付形态 | 参考区间(¥) | 说明 |
|---|---|---|
| 图文教程/清单 | 9.9-29 | 只适合引流,不是主力产品 |
| 远程代装/一次性部署 | 49-199 | 按复杂度;含一次答疑 |
| 汉化包/行业模板包 | 29-99 | 要承诺跟随上游更新 |
| 部署+按月维护 | 99-299/月 | 最稳的复购形态 |
| 定制开发/集成 | 300+ | 需要技术能力,按项目报价 |

报告里给区间和理由,最终定价由用户人工验证竞品后确定。

## 单仓模式输出

只输出:四查评分卡 + 结论(直接试/需要技术基础/观望/关掉)+ 理由(每查一句话)+ 若有服务机会则附机会四问答案并写台账。不生成 HTML(除非用户要)。

## 收尾必做

- 报告末尾列出**本次运行的局限**:哪些访问方式被降级、候选覆盖是否受限、哪些结论依赖人工验证——沉默的截断会让"看过了"变成假象。
- 提醒用户台账里 `待验证` 状态的机会数量,以及验证动作(人工闲鱼抽查清单在报告里)。
