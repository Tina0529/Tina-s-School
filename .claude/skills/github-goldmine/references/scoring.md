# 五分钟判断法:评分细则 + Issues 需求挖掘方法

## 第一部分:四查评分

对每个候选项目按四个维度打分(每维 0-5 分),总分 20。打分不是目的,目的是快速把项目分进四个桶。

### 查 1:README(它到底做什么)

| 分数 | 信号 |
|---|---|
| 4-5 | 一句话说清用途 + 截图/演示地址 + 明确的系统支持说明 + 安装步骤面向普通用户 |
| 2-3 | 用途清楚但安装说明只有终端命令;或有截图但没说支持什么系统 |
| 0-1 | 只有宏大口号没有使用示例;或 README 基本是给开发者看的 API 文档 |

### 查 2:Releases(有没有成品)

| 分数 | 信号 |
|---|---|
| 4-5 | 有正式版本,含对应系统安装包(.exe/.msi/.dmg/AppImage),区分了稳定版和预发布版 |
| 2-3 | 有版本但只有 Docker 镜像/pip 包/需要命令行;或安装包只覆盖部分系统 |
| 0-1 | Releases 为空,或只有自动生成的 Source code.zip |

注意:**2-3 分正是服务机会区**——项目有价值但作者没替普通用户做完最后打包。

### 查 3:维护状态(现在还活着吗)

| 分数 | 信号 |
|---|---|
| 4-5 | 90 天内有正式发布或活跃提交,版本说明写清修了什么 |
| 2-3 | 半年内有更新;或离线小工具虽久未更新但功能自洽、不依赖外部接口 |
| 0-1 | 已归档(Archived);或依赖外部接口/AI 模型/浏览器却半年没动;或满屏"现在还能用吗"无人回应 |

时间不能一刀切:离线工具半年不更新可能照常能用;依赖外部环境的工具半年没动就要警惕。

### 查 4:Issues(用户卡在哪、有没有人管)

| 分数 | 信号 |
|---|---|
| 4-5 | 维护者追问复现方式、补文档、给临时方案,问题最终有修复或明确说不修 |
| 2-3 | 有回应但慢;或问题多但集中在安装(说明有价值但上手难) |
| 0-1 | 同类报错反复出现长期无人回应;或 Issues 被关闭不让提 |

### License 检查(不打分,直接分流)

- **MIT / Apache-2.0 / BSD**:商用友好,保留版权声明即可 → 全部服务形态可行
- **GPL / AGPL**:卖部署、教学、维护服务不受限;分发修改版必须开源;AGPL 连托管服务都要开源修改 → 标注清楚,服务类机会仍可行
- **CC-NC / 自定义非商用条款**:排除出机会列表,只能作为免费内容选题
- **无 License**:默认保留所有权利,不能商用 → 排除

### 四桶结论

- **直接试**(≥14 且查 2 ≥ 3):用途清楚、有成品、有人维护
- **需要技术基础**(≥10 且查 2 ≤ 3):有价值但只有源码/容器/终端 → **服务机会重点区**
- **观望**(7-9 分):版本早、同类问题多、维护不稳
- **关掉**(≤6,或触发红旗):说明混乱、长期停更、无人管,或索要重要账号权限/密钥却解释不清

红旗(出现即"关掉",不看总分):要求提供主账号登录态/钱包/支付信息且无解释;社区有实名的安全质疑无人澄清;仓库是无来源的搬运号。

## 第二部分:Issues 需求挖掘方法

### 抱怨四分类

先把扫到的 Issues 分成四类,每类指向不同机会:

| 类别 | 典型问题 | 指向的机会 |
|---|---|---|
| 安装问题 | "Windows 怎么装?" "照着 docker 示例跑不起来" | 代装、打包、图文教程 |
| 选择问题 | "它和 X 有什么区别?" | 对比评测内容、选型咨询 |
| 功能问题 | "能导出表格吗?" "能同步网盘吗?" | 插件、模板、数据迁移服务 |
| 信任问题 | "数据存在哪?" "会发给外部接口吗?" | 私有化部署服务、安全配置指导 |

### 信号强度判断(从弱到强)

1. 单条留言、无人附议 → 记录但不立项
2. 同一项目内同类问题反复出现、有 +1 和 👍 → 候选
3. 用户已付出成本:手动整理、反复重装、自己写脚本绕过 → 强信号(愿意忍受笨办法的人才愿意付钱省事)
4. **跨项目重复**:同类抱怨在 3 个以上同领域项目出现 → 最强信号,这是一类人的共同麻烦,不是个例

### 维护者态度的解读

- 修复了 → 需求已被满足,机会关闭(但旧版本用户的迁移可能是新机会)
- 承诺做但拖了几个月 → 时间窗口:先做服务占位
- `wontfix` / `not planned` → **最佳缺口**:不符合原项目定位,但用户仍在追问,原作者永远不会来竞争
- 沉默 → 结合维护状态判断:活跃项目的沉默是优先级问题,停更项目的沉默是机会(接盘维护/迁移方案)

### 站内搜索语法(API/gh 可用时)

```
repo:owner/name export is:issue           # 查导出需求
repo:owner/name 中文 OR chinese is:issue   # 查中文需求
repo:owner/name install is:issue sort:comments-desc  # 按讨论热度看安装问题
```

WebFetch 兜底时,直接抓 `https://github.com/owner/name/issues?q=<关键词>` 页面提取。
